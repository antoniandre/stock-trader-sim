## 4 improvements to push PF from 1.22 → 1.5+

### 1. Volatility-regime switch (biggest lever — addresses Jan-Feb problem directly)

**Problem**: The trend setup fails in high-volatility market regimes (37–43% WR in Jan-Feb 2026 vs 56% WR in March-April). All the existing hard gates couldn't filter this because the condition is at the **market level**, not the individual stock level.

**Solution**: Build a rolling SPY volatility indicator that the sim computes per-bar and passes as `input.marketVolatileRegime: boolean`. When elevated, suppress trend entries via a new hard gate.

**How to compute in `scripts/run-100-trades.mjs`**:

```javascript
// buildVolatilityRegimeMap: SPY 5-day rolling ATR as % of close.
// When SPY ATR-5day > 1.2× its own 20-day average = volatile regime.
function buildVolatilityRegimeMap(spyCandles) {
  const map = new Map()  // timestamp → boolean (true = volatile)
  for (let i = 20; i < spyCandles.length; i++) {
    // 5-day ATR: last 5 bars' true ranges averaged
    let atr5 = 0
    for (let k = i - 4; k <= i; k++) {
      const h = spyCandles[k].high, l = spyCandles[k].low, pc = spyCandles[k - 1].close
      atr5 += Math.max(h - l, Math.abs(h - pc), Math.abs(l - pc))
    }
    atr5 = (atr5 / 5) / spyCandles[i].close * 100

    // 20-day ATR baseline (average of 20 daily true ranges, each day = 78 5-min bars)
    // Simplified: use the per-bar ATR of the last 20 bars as a proxy
    let atr20 = 0
    for (let k = i - 19; k <= i; k++) {
      const h = spyCandles[k].high, l = spyCandles[k].low, pc = spyCandles[k - 1].close
      atr20 += Math.max(h - l, Math.abs(h - pc), Math.abs(l - pc))
    }
    atr20 = (atr20 / 20) / spyCandles[i].close * 100

    // Volatile if 5-day ATR is 30%+ above the 20-day baseline
    map.set(spyCandles[i].timestamp, atr5 > atr20 * 1.3)
  }
  return map
}
```

Pass `marketVolatileRegime: volRegimeMap.get(c.timestamp) ?? false` in the decision call.

**Gate in `api/day-trading-bot.js`** (add after `circuitBreakerActive` gate):

```javascript
const marketVolatileRegime = Boolean(input.marketVolatileRegime ?? false)

// In buy-conditions chain, after circuit breaker:
else if (setup === 'trend' && marketVolatileRegime) {
  action = 'wait'
  reasons.push('Volatility-regime gate: SPY ATR is elevated — suppressing trend entries until market calms')
}
```

**Expected impact**: The Jan-Feb period where trend WR was 37% would have most trend entries blocked. Trend WR should rebound to the 55%+ seen in trending conditions.

### 3. VWAP reclaim as a positive breakout signal

**Problem**: The current VWAP logic only has a negative gate (block when price is > 0.2% below VWAP). It doesn't detect one of the highest-probability intraday setups: **price reclaiming VWAP after a pullback**. This is a classic institutional re-entry signal.

**Solution**: Track whether price crossed VWAP from below (reclaim) within the last 3 bars, and if so, add it as an entry boost and potentially as a new "vwap-reclaim" setup classification.

**How to implement in `scripts/run-100-trades.mjs`**:

```javascript
// Was price below VWAP 3 bars ago and now above it?
const prevVwapDev = prevPrice > 0 && vwapVal > 0
  ? ((prevPrice - vwapVal) / vwapVal) * 100
  : 0
const vwapReclaim = prevVwapDev < -0.1 && vwapDevPct > 0.1  // crossed from below to above
```

Pass `vwapReclaim: vwapReclaim` in the decision call.

**In `api/day-trading-bot.js`**:
- Add `const vwapReclaim = Boolean(input.vwapReclaim ?? false)`
- In entry scoring: `if (vwapReclaim && setup !== 'mean-revert') entryScore += 10` — significant boost because VWAP reclaim is a directional confirmation signal
- Optionally: add a new gate that **requires** VWAP reclaim for trend entries (replacing the ORB gate, or in addition to it). A trend entry above VWAP that just reclaimed it = high institutional conviction.

**Expected impact**: Filters trend entries where price has been above VWAP for a while (extended, distribution risk) in favor of fresh reclaims (institutional re-entry). Should improve trend WR by ~5-8%.

### 4. Per-setup exit tuning (separate breakout from trend exit logic)

**Problem**: Breakout and trend setups have fundamentally different holding characteristics but currently share the same exit rules. Specifically:
- Breakout winners should be held until the 2.4% target or trailing stop — the move has conviction
- Trend winners at 1R gain with pausing momentum are a coin flip — trim is appropriate

Currently the 1R partial trim (`shortTrendPct < -0.15`) applies to both. Breakout trades that pause at 1R (trend turns slightly negative) get trimmed even when they should be held.

**Solution**: Track the `setup` at entry time and use it to gate the 1R partial trim.

**In `scripts/run-100-trades.mjs`**, store entry setup:

```javascript
entryIndicators = {
  // ... existing fields ...
  entrySetup: decision.setup  // add this
}
```

Pass it back on each bar while in position:

```javascript
const decision = evaluateDayTradingDecision({
  // ... existing inputs ...
  entrySetup: entryIndicators?.entrySetup ?? null
})
```

**In `api/day-trading-bot.js`**, read the entry setup:

```javascript
const entrySetup = input.entrySetup ?? null  // setup at the time of entry, not current bars
```

Modify the 1R partial exit:

```javascript
// Skip 1R trim on original breakout entries — let them run to the 2R target or trailing stop.
// Only trim trend entries that have paused, since their continuation rate is lower.
else if (
  unrealizedPnLPct >= dynamicStopPct
  && !takeProfitZone
  && shortTrendPct < -0.15
  && entrySetup !== 'breakout'
) {
  action = 'trim'
  reasons.push(`1R partial exit: ${round(unrealizedPnLPct)}% gain with fading momentum — locking partial profit`)
}
```

**Expected impact**: Breakout winners that currently get trimmed at 1R will instead run to 2R or the trailing stop, increasing avg win from ~$94 to ~$130–150. Combined with 63–73% WR, this should push PF to 1.6–1.8.
